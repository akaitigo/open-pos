# サービスメッシュ評価 (#206)

## 概要

open-pos のマイクロサービスアーキテクチャに対するサービスメッシュ導入の検討。

## 候補

### Istio

| 項目 | 評価 |
|------|------|
| 機能の豊富さ | 高（トラフィック管理、mTLS、可観測性、ポリシー適用） |
| 学習コスト | 高（Envoy proxy + 多数の CRD） |
| リソース消費 | 高（sidecar + コントロールプレーン） |
| GKE 統合 | Anthos Service Mesh として Google マネージド版あり |
| コミュニティ | 最大手。CNCF graduated |

**メリット**:
- GKE との親和性（Anthos SM でコントロールプレーン管理不要）
- 高度なトラフィック制御（カナリア、フォールトインジェクション）
- JWT 検証やレートリミットを mesh レベルで実装可能

**デメリット**:
- メモリ消費: 各 Pod に Envoy sidecar (~50MB)
- 設定の複雑さ（VirtualService, DestinationRule, AuthorizationPolicy 等）
- 小規模構成ではオーバーエンジニアリング

### Linkerd

| 項目 | 評価 |
|------|------|
| 機能の豊富さ | 中（mTLS、可観測性、リトライ。トラフィック制御は Istio より簡素） |
| 学習コスト | 低（設計がシンプル） |
| リソース消費 | 低（Rust ベースの micro-proxy: ~10MB/pod） |
| GKE 統合 | 公式サポートなし（自前運用） |
| コミュニティ | CNCF graduated。Istio より小さい |

**メリット**:
- 軽量: sidecar のリソース消費が少ない
- シンプル: 設定が少なく導入障壁が低い
- Kubernetes ネイティブ

**デメリット**:
- GKE マネージド版がないため自前運用
- 高度なトラフィック制御（ヘッダーベースルーティング等）が弱い
- Ingress Gateway 機能が限定的

## open-pos への適用

### 現状の構成
- 6 マイクロサービス + 1 API Gateway
- gRPC によるサービス間通信
- Cloud Run / GKE でのデプロイ

### 推奨

**フェーズ 1（現状）**: サービスメッシュは不要
- 6 サービス規模では K8s Service + gRPC で十分
- API Gateway が認証・レートリミット・ルーティングを担当
- mTLS は gRPC + TLS 設定で対応可能

**フェーズ 2（サービス数 10 以上）**: Linkerd を推奨
- 軽量で導入が容易
- mTLS の自動化で運用負荷軽減
- 可観測性（Grafana 統合）で障害調査を効率化

**フェーズ 3（エンタープライズ）**: Istio / Anthos SM を推奨
- GKE マネージドでコントロールプレーン管理不要
- 高度なトラフィック制御が必要な場合
- マルチクラスタ・マルチリージョン展開時

## 結論

現時点ではサービスメッシュの導入は見送り。サービス数が 10 を超えた時点で Linkerd の導入を検討する。
