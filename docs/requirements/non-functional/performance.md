# パフォーマンス要件

## レスポンスタイム目標

| カテゴリ | 目標値 | 測定条件 |
|---------|--------|---------|
| REST API（通常） | p95 200ms以内 | 正常系、単一リクエスト |
| 取引確定（カート→レシート） | 1秒以内 | e2eレイテンシ |
| オフライン操作（UI） | 100ms以内 | IndexedDB読み書き含む |
| マスタ同期（差分） | 2秒以内 | 1000件更新差分 |
| バーコードスキャン検索 | 100ms以内 | ローカルキャッシュ検索 |

## スループット目標

| シナリオ | 目標値 |
|---------|--------|
| 同時接続端末数（1店舗） | 10端末 |
| 取引処理（1店舗・ピーク） | 30件/分 |
| 全テナント合計スループット | 水平スケールで対応 |

## サービス別SLO

| サービス | 可用性 | レイテンシ(p95) |
|---------|--------|----------------|
| api-gateway | 99.9% | 50ms |
| pos-service | 99.9% | 100ms |
| product-service | 99.5% | 50ms |
| inventory-service | 99.5% | 100ms |
| analytics-service | 99.0% | 500ms |

## キャッシュ戦略

| データ | キャッシュ層 | TTL |
|--------|------------|-----|
| 商品マスタ | Redis | 5分 |
| 税率 | Redis | 1時間 |
| 店舗設定 | Redis | 10分 |
| 取引（読み取り） | キャッシュなし（常にDB） | - |

- cache-aside パターン（Read: キャッシュ→ミス→DB→キャッシュ書き込み）
- 全 Redis キーに TTL 必須（TTL なしキー禁止）

## データベース

- pgBouncer でコネクションプール（pool_size: 20/service）
- N+1クエリ禁止（JOIN FETCH または バッチフェッチ）
- 重要クエリには実行計画確認（`EXPLAIN ANALYZE`）必須

## 監視指標

- API レスポンスタイム（p50/p95/p99）
- DB クエリ時間
- RabbitMQ キュー深度（遅延検知）
- Redis ヒット率
- エラーレート（5xx）

## 受け入れ条件

- [ ] 取引確定が1秒以内（ステージング環境で負荷テスト済み）
- [ ] 10端末同時アクセスでレスポンスタイム劣化が20%以内
- [ ] p95 200ms を k6 負荷テストで確認
