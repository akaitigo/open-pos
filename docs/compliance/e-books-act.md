# 電子帳簿保存法 対応ガイド

## 概要

電子帳簿保存法（電帳法）は、国税関係帳簿書類の電子保存に関する法律である。2024年1月に電子取引データの電子保存が完全義務化され、POS システムで発生する電子取引データ（レシート、請求書、領収書等）は電子的に保存する義務がある。

### 3つの保存区分

| 区分 | 内容 | open-pos での該当 |
|------|------|------------------|
| 電子帳簿等保存 | 自己が作成した帳簿・書類を電子保存 | 売上台帳、在庫台帳 |
| スキャナ保存 | 紙で受領した書類をスキャンして保存 | 仕入先からの紙の納品書（対象外） |
| 電子取引データ保存 | 電子的に授受した取引データを保存 | 電子レシート、電子請求書 |

## 電子取引データの保存要件

### 真実性の確保

以下のいずれかの措置が必要:

1. タイムスタンプ付与後に取引データを授受
2. 授受後速やかにタイムスタンプを付与（最長約2ヶ月＋概ね7営業日以内）
3. 訂正削除ができないシステム、または訂正削除の事実・内容を確認できるシステムを使用
4. 訂正削除の防止に関する事務処理規程を定めて運用

### 検索要件

以下の検索条件で取引データを検索できること:

- **取引年月日**: 日付範囲での検索
- **取引金額**: 金額範囲での検索
- **取引先名称**: 取引先での検索

> 基準期間の売上高が5,000万円以下の事業者は、税務調査時にデータのダウンロードに応じる場合、検索要件が不要。

### 可視性の確保

- ディスプレイ・プリンタ等で整然とした形式で出力できること
- システム仕様書（マニュアル）を備え付けること

## open-pos での対応状況

### 対応済み（真実性の確保 — 措置3に該当）

open-pos は PostgreSQL + Flyway による不変レコード設計を採用しており、取引データの訂正削除の事実を確認できる。

#### データの不変性

```sql
-- 取引テーブルの設計（pos_schema.transactions）
-- completed_at にタイムスタンプを記録し、レコードは物理削除しない（論理削除のみ）
-- 取消は VOID トランザクションとして新規レコードを作成

SELECT
    t.id,
    t.organization_id,
    t.transaction_number,
    t.total,                 -- 銭単位 BIGINT
    t.completed_at,          -- 取引完了タイムスタンプ（UTC）
    t.created_at,
    t.type,                  -- SALE / RETURN / VOID
    t.content_hash           -- SHA-256ハッシュ（真正性確保）
FROM pos_schema.transactions t
WHERE t.completed_at BETWEEN '2026-01-01' AND '2026-12-31'
  AND t.organization_id = :orgId
  AND t.deleted = false;
```

#### 監査ログ

```sql
-- store_schema.audit_logs で全変更を追跡
SELECT
    action,
    entity_type,
    entity_id,
    staff_id,
    created_at,
    details,
    ip_address
FROM store_schema.audit_logs
WHERE entity_type = 'TRANSACTION'
ORDER BY created_at DESC;
```

#### Flyway マイグレーション

- スキーマ変更は全て Flyway マイグレーションで管理
- マイグレーション履歴は `flyway_schema_history` テーブルに記録
- 適用済みマイグレーションの改ざんはチェックサムで検出

### 対応済み（検索要件）

| 検索条件 | 対応方法 |
|---------|---------|
| 取引年月日 | `completed_at` カラムにインデックス、日付範囲検索対応 |
| 取引金額 | `total` カラムで金額範囲検索対応 |
| 取引先名称 | `organization_id` + 顧客情報で検索対応 |

管理画面（admin-dashboard）から上記条件での検索・CSV エクスポートが可能。

### 対応済み（可視性の確保）

- 管理画面からの帳票表示・印刷機能
- 本ドキュメントおよび `docs/` 配下のシステム仕様書

## 未対応事項と今後の計画

### P1: 認定タイムスタンプの導入

現在はデータベースのタイムスタンプ（`created_at`, `completed_at`）を使用しているが、一般財団法人日本データ通信協会の認定タイムスタンプ局による認定タイムスタンプの付与を検討する。

- **対応方針**: 外部タイムスタンプサービス（例: アマノタイムスタンプサービス）との連携
- **優先度**: 中（現行の措置3で法的要件は満たしている）

### P2: 事務処理規程の整備

措置4に基づく事務処理規程のテンプレートを整備する。

- 電子取引データの保存・管理に関する社内規程
- 訂正削除のワークフローと承認プロセス

### P3: 長期保存とアーカイブ

- 法定保存期間（7年、繰越欠損金がある場合は10年）に対応したアーカイブ戦略
- コールドストレージ（S3 Glacier 等）への自動アーカイブ
